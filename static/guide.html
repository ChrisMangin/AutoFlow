What is AutoFlow?

AutoFlow is a desktop automation tool that lets you record mouse & keyboard actions, edit them into reliable workflows, and play them back hands-free. No scripting required for most tasks — just record, review, and run.

Workflows are stored as plain JSON files so they're portable and version-controllable. AutoFlow runs entirely locally — nothing leaves your machine.

ℹ️

AutoFlow lives in your system tray when the main window is closed. Double-click the tray icon or right-click it to reopen, record a new workflow, or replay the last one.

Recording

Recording captures every click, keystroke, and scroll you make across your entire desktop — including inside desktop apps, browsers, file explorers, and terminals.

Basic recording flow

  1. Click ● Record in the toolbar (or press the tray hotkey)
  2. Perform the actions you want to automate
  3. Click ■ Stop (or close the action) when done
  4. Review the captured steps in the canvas, edit or delete any you don't need
  5. Name the workflow in the title bar and click 💾 Save
💡

Pause during recording — hit ⏸ Pause to freeze capture while you navigate to the right window, then resume. Paused time is not recorded.

What gets captured

  • Mouse clicks (with target window, element name when available, and screen coordinates as fallback)
  • Keyboard typing — captured as a single Type step per burst
  • Hotkeys — Ctrl+C, Alt+F4, etc.
  • Mouse wheel scrolls — consecutive scrolls at the same position are merged into one step automatically

Automatic cleanup on stop

When you stop recording, AutoFlow condenses the raw event stream:

  • Navigational scrolls removed — if you scrolled through a dropdown or list to find an item and then clicked it, the scroll steps are dropped. The click stores the scroll direction so the player can scroll-and-search visually at playback time, without needing exact scroll counts.
  • Repeated scrolls merged — consecutive scroll events at the same position become one step with the total delta.
  • Opening preamble collapsed — leading clicks that were just "getting to the application" (opening Start menu, navigating to a page) are replaced with a single Navigate + Wait for Window step.
  • Duplicate waits merged — back-to-back wait steps keep only the longer one.
⚠️

Chrome element targeting — Chrome's accessibility layer is off by default. For most browser clicks, AutoFlow automatically falls back to image-based matching: it captures the exact visual region around your click (while the dropdown or menu is still open) and re-locates it visually at playback time. This handles dynamic layouts and dropdown menus without any browser flags. For the most reliable element-level targeting, add a 🚀 Launch Browser step at the top of your workflow — it opens Chrome with accessibility enabled so AutoFlow can also target by element name.

Playback

Full playback

Click ▶ Play. AutoFlow replays every step from the beginning (or from your chosen start point — see below). The active step is highlighted in the canvas as it runs.

Start from a specific step

Click any step card to set it as the playback start point — a green indicator appears at the bottom. ▶ Play and ⏭ Step both respect this starting point.

Step-through mode

Click ⏭ Step to execute one step at a time. Useful for debugging — you can see exactly what each step does before continuing.

Pause & resume

Hit ⏸ Pause during playback to freeze at the current step. Click ▶ Resume to continue.

Playback speed

The Speed slider in the toolbar scales all wait times. 0.25× slows everything down for debugging; 4× races through fast repetitive tasks.

Element targeting

The Element targeting checkbox (toolbar) tells playback to use captured UI element names rather than raw screen coordinates. Leave it on unless an element-based step is failing — then try unchecking it to fall back to coordinates.

Saving & Loading

  • Save — writes dist/workflows/<name>.json. The name comes from the titlebar field.
  • Load — click any workflow in the Workflows sidebar section.
  • Delete — hover a workflow name and click the red ✕ that appears.
  • New — clears the canvas for a fresh workflow without closing the app.

Workflow JSON files are portable — copy them between machines, put them in Git, or share them with colleagues who also run AutoFlow.

Variables

Variables are named values stored in the right panel. They let you parameterize workflows so the same steps work with different inputs.

Creating variables

Click ＋ Variable in the right panel. Give it a name (letters, numbers, underscores) and an optional default value.

Using variables in steps

Anywhere a step field accepts text, wrap the variable name in curly braces: {my_variable}. AutoFlow substitutes the live value at playback time.

# Type step: logs into a form
value: Hello {username}, your ID is {user_id}

Setting variables during playback

  • Set Variable — assign a literal or expression
  • Get Clipboard — reads clipboard into a variable
  • Read File — reads file content into a variable
  • HTTP Request — stores the response body
  • Run Script — set ctx["varname"] = value to write any variable

Logic & Loops

Loop

The 🔁 Loop step repeats the steps between it and its matching ↩️ End Loop marker N times. Set count to the number of iterations.

🔁 Loop  (count: 5)
  🖱️ Click  ...
  ⌨️ Type   ...
↩️ End Loop

If / Else

The ❓ If step branches based on a variable value. Set variable, operator (== != > < contains), and value.

❓ If  (status == "done")
  📢 Show Message: "Finished!"
↕️ Else
  📢 Show Message: "Still running…"
✓  End If

Run Script

The ⚙️ Run Script step executes an inline Python snippet inside the workflow runtime. Use it for anything the built-in steps can't do.

The ctx object

ctx is a dict containing all current variable values. Read from it, write to it:

# Double a numeric variable
ctx["count"] = int(ctx.get("count", 0)) * 2

# Build a filename from variables
ctx["output_path"] = f"C:/reports/{ctx['date']}_report.txt"
ℹ️

Scripts run in the same Python process as AutoFlow. Standard library imports work. Third-party packages available in AutoFlow's venv are also accessible.

Export & Import

PDF Report — 📄 PDF

Generates a formatted PDF of the entire workflow: step list, parameters, step notes (shown as green callout blocks), and any screenshots captured by 📸 Screenshot steps. Disabled steps are shown with a "SKIPPED" tag so reviewers can see what was intentionally bypassed.

Python Script — 🐍 Script

Exports the workflow as a standalone .py file. The script runs without AutoFlow installed — it uses pyautogui directly. Good for scheduling via Task Scheduler or including in larger codebases.

Runnable Package — 📦 ZIP

Exports a .zip file containing the Python script, a run.bat launcher, requirements.txt, and a README.txt. Share this ZIP with anyone on Windows — they just double-click run.bat. The batch file checks whether Python is installed, installs pyautogui automatically if needed, then runs the workflow. No AutoFlow installation required on the recipient's machine.

ℹ️

Recipients who don't have Python will see clear instructions pointing them to python.org/downloads. Make sure they check "Add Python to PATH" during installation.

Import — 📥 Import

The 📥 Import button in the titlebar accepts two file types:

  • .json — a saved AutoFlow workflow file. Loads it directly as the current workflow.
  • .py — a Python script previously exported from AutoFlow. AutoFlow parses it back into editable steps (click, type, hotkey, wait, scroll, navigate, loop, if/else, variables, and comments all round-trip correctly).

Tips & Tricks

Editing steps

Double-click any step card to open the edit modal. You can change every field — coordinates, text, timeout, window title, etc. Click the card's ✕ button to delete it.

Step notes

Every step card has a 💬 Add note… area at the bottom. Click it and type anything — a reminder, a description of what the step does, a reason why it's there. The note saves automatically when you click away or press Enter. Press Shift+Enter for a line break; Esc cancels without saving.

The same note field appears at the bottom of the step's edit modal (📝 Note) if you prefer editing there.

Notes show on the card in italic and appear as a 📝 Note callout in the PDF export — making the exported PDF useful as a walkthrough document or SOP if you choose to write it that way.

Reordering steps

Drag and drop cards to reorder them. The table view shows the same steps in a compact list — changes in one view are reflected in the other.

Disabling steps

Click the 🚫 button on any step card to disable it without deleting it. Disabled steps are visually greyed out, skipped during playback, and commented out in exported scripts. Click ▶ to re-enable. This is useful when debugging a workflow — disable a step temporarily to see what happens without it.

Undoing a deleted step

Pressing Ctrl+Z immediately after deleting a step restores it to its original position. AutoFlow keeps the last 10 deleted steps in memory — so you can undo up to 10 deletions in sequence. A brief toast notification appears after each deletion reminding you the shortcut is available.

Unsaved changes indicator

The Save button pulses blue and the page title shows ● whenever the current workflow has unsaved edits — including adding, deleting, reordering, editing, or disabling steps. The indicator clears as soon as you save.

Start from any step

Click a card once to mark it as the playback start point (highlighted green dot in status bar). Hit ▶ Play and only that step onward runs — useful when iterating on the end of a long workflow.

Script export — supported step types

The 🐍 Script and 📦 ZIP exports generate standalone Python scripts using pyautogui. The following step types are fully supported in exported scripts:

  • Core input: Click, Type, Hotkey, Scroll, Wait, Navigate
  • Flow control: Loop, If/Else, Set Variable, Run Script, Comment
  • System: Open File/App (os.startfile), Kill Process, Close Window, Play Sound
  • Files: Read File, Write File, Copy File, Move File, Delete File
  • Network: HTTP Request (urllib, no extra dependencies)
  • UI: Wait for Window, Launch Browser, Show Message, Get/Set Clipboard, Screenshot

Steps that are disabled are written as comments. Image Click steps include a comment showing how to locate the template image using pyautogui.locateCenterOnScreen.

The exported script auto-installs pyautogui on first run if it is not already installed.

Chrome web automation

  1. Add a 🚀 Launch Browser step as step 1
  2. Add a 🌐 Navigate step with your target URL
  3. Record or add the remaining interaction steps

This ensures Chrome opens with accessibility enabled and AutoFlow can reliably identify buttons, inputs, and links by name rather than pixel position.

Handling dynamic content

Use ⏳ Wait for Element or 🪟 Wait for Window before clicking things that load asynchronously. Avoid raw ⏱️ Wait steps with fixed times — they're fragile if a page loads faster or slower than expected.

Dropdown and list navigation

When you scroll through a dropdown or list to reach an item, AutoFlow records the scroll amount alongside the click. At playback it:

  1. Brings the target window to front
  2. Moves the mouse to the recorded click position
  3. Replays the scroll — the same number of notches in the same direction as during recording — so the item is back at the click position
  4. Clicks the item

This means dropdown selections work correctly even when the dropdown opens at a different scroll position, as long as the current selection at playback time is similar to what it was during recording (so the same scroll offset reaches the same item).

Tip: If an item is consistently missed, the dropdown may open at a very different scroll position on the playback machine. Re-record the selection from that starting state.

Using variables for portability

Replace hardcoded paths and usernames with variables. Set the variable once at the top of the workflow with 📦 Set Variable, then every step that needs it uses {varname}. Changing the workflow for a new target means editing one field.

Sidebar sections

Click any sidebar section header to collapse or expand it. Your preference is saved and restored every time you open the app. By default, only the Workflows list is expanded — the step-building palette sections are collapsed so the sidebar stays tidy.

Multi-monitor setup

The recording HUD (the floating status indicator) automatically repositions itself to the monitor your cursor is on each time recording starts. If you use different monitor arrangements, you don't need to do anything — it will follow you.

For screenshot steps and recorded click previews, use the ⚙️ Settings sidebar section to choose the capture mode:

  • All screens (default) — captures the entire virtual desktop spanning all monitors. Best when you want full context across screens.
  • Active screen only — captures only the monitor where the action occurred. More accurate on HiDPI or mixed-DPI setups where coordinates on secondary monitors can be offset.

Troubleshooting

SymptomFix
Clicks land in the wrong spotFor browser/web clicks, AutoFlow uses image matching to find the element regardless of window position. If it still fails, the visual layout may have changed significantly — re-record that step. For native app clicks, ensure the window is open and in a similar position to when you recorded.
"NativeViewHost" warning bannerAdd a 🚀 Launch Browser step at the top so Chrome opens with accessibility enabled.
Playback races too fast / too slowAdjust the Speed slider. Add ⏱️ Wait steps after actions that take time to complete.
Dropdown item not selected correctly during playbackAutoFlow replays the recorded scroll before clicking. If the wrong item is selected, the dropdown may have opened at a very different scroll position on the playback machine (e.g. current selection is far from the recorded one). Re-record the dropdown selection starting from the same site/state the playback machine will be in.
Step fails on a renamed windowEdit the step's window field — AutoFlow does case-insensitive partial title matching, so you only need a stable substring (e.g. Excel instead of Book1 - Excel).
Exported script doesn't open Excel / appOlder exported scripts had # TODO comments for open_file and wait_for_window steps. Re-export the script — these are now fully supported.
Wait for Window times out (Excel, Office)The default timeout is 8 seconds. Increase it via the step's timeout_ms field (e.g. 20000 for 20 s). Also use a short substring like Excel as the title — Office apps sometimes open with a splash screen before showing the workbook title.
Open File/App step fails (Excel, Word, etc.)AutoFlow first tries os.startfile, then automatically falls back to cmd /c start which searches the Windows App Paths registry. This means excel.exe, winword.exe, and other Office apps work without specifying the full path. If an app still fails to open, provide its full path (e.g. C:/Program Files/...) in the step's value field.
Variable substitution not workingMake sure the variable name in the step matches exactly (case-sensitive). Check the Variables panel to confirm the variable exists.
App won't quitClick ✕ Quit in the titlebar, or right-click the tray icon and choose Quit. As a last resort, end the AutoFlow.exe process in Task Manager.
Workflow file is corruptWorkflow JSON files are in dist/workflows/. Open the file in a text editor to inspect or fix it. AutoFlow will skip malformed files on load.
💡

Check dist/autoflow.log for detailed error messages if something unexpected happens during playback.
